TLS Decryption Without TLS Termination

Unlike proxies or sidecars that must terminate TLS to inspect traffic, the eBPF sniffer hooks into SSL_read() and SSL_write() at the library level. It reads the plaintext before encryption and after decryption — completely transparent to the application and the network. No certificates to manage, no MITM, no latency added to the request path.

Core Technology

eBPF SSL/TLS Sniffer

A kernel-level packet interception tool that captures plaintext HTTP from encrypted TLS connections. It uses eBPF uprobes to hook into SSL library functions at runtime — extracting request and response data before encryption and after decryption.

🔍

Automatic Library Discovery

Scans /proc/*/maps at startup to find every loaded SSL library across all running processes — including those inside containers. Discovers OpenSSL, GnuTLS, NSS, Go crypto/tls, and Rust rustls automatically.

⚙

eBPF Uprobe Interception

Attaches uprobes to SSL_read, SSL_write, SSL_read_ex, and SSL_write_ex functions. Captures the plaintext buffer content, return values, process metadata, and timing information.

🔁

Request/Response Correlation

Matches SSL write (request) and read (response) events from the same PID+TID into complete HTTP transactions. Supports both client-side and server-side capture perspectives.

📡

Telemetry Pipeline

Correlated HTTP pairs are serialized into Reconstructor-compatible telemetry format and sent via HTTP POST with retry logic, exponential backoff, and circuit breaker protection.

🛡

Security & Redaction

Sensitive headers (Authorization, Cookie, API keys) are automatically redacted. PCI-DSS patterns in bodies are masked. Audit logging tracks all captured events.

📊

Observability

Prometheus metrics on port 9090, health/readiness probes on port 8080. Tracks events captured, lost, latency, correlator state, and output circuit breaker status.

How eBPF Uprobes Work

// 1. Application calls SSL_write() to send encrypted data
// 2. eBPF uprobe fires BEFORE the function executes
//    → captures: buffer pointer, process ID, thread ID, timestamp

int probe_SSL_rw_enter(struct pt_regs *ctx, void *ssl, void *buf, int num) {
    u32 tid = bpf_get_current_pid_tgid();
    bufs.update(&tid, &buf);          // Store buffer pointer
    start_ns.update(&tid, &ts);       // Store timestamp
    return 0;
}

// 3. SSL_write() executes normally (encrypts and sends)
// 4. eBPF uretprobe fires AFTER the function returns
//    → reads plaintext from stored buffer, emits event

static int SSL_exit(struct pt_regs *ctx, int rw) {
    int len = PT_REGS_RC(ctx);        // Bytes read/written
    bpf_probe_read_user(&data->buf, len, *bufp);  // Copy plaintext
    perf_SSL_rw.perf_submit(ctx, data, size);      // Emit to userspace
    return 0;
}
Highlight

Full Request & Response Payload Capture

The sniffer doesn't just capture headers — it captures complete HTTP request and response bodies including JSON payloads, form data, and file uploads up to a configurable size (default 8KB per SSL call, max 64KB). This means the Reconstructor can analyze request body schemas, response structures, parameter types, and error formats to build richer OpenAPI specifications. Sensitive fields (credit card numbers, tokens) are automatically redacted before transmission.

Highlight

OpenSSL 3.x Extended Function Support

Modern applications (Python 3.12+, Node.js) call SSL_read_ex/SSL_write_ex which return 1/0 (success/failure) instead of the byte count. The actual bytes are written to a 4th pointer parameter. The sniffer includes dedicated BPF handlers that dereference this pointer to extract the real byte count — a nuance that most eBPF SSL sniffers miss entirely.

Technical Details

Under the Hood

🧬

BPF Program Compilation

The eBPF C program is compiled at runtime by BCC with kernel-specific filters injected (PID, UID, comm, self-exclusion). Falls back from ring buffers to perf buffers on older kernels.

🔄

SSL_read_ex / SSL_write_ex

OpenSSL 3.x extended functions use a 4th pointer parameter for byte count (return value is 1/0). Dedicated BPF handlers dereference this pointer via bpf_probe_read_user.

🔀

Server-Side Correlation

The correlator handles both perspectives: client-side (WRITE=request) and server-side (READ=request). Uses parsed HTTP content to determine role, not event direction.

📦

Per-CPU Lock-Free Buffers

BPF_PERCPU_ARRAY ensures zero contention between CPUs. Events flow through perf buffers (or ring buffers on 5.8+) without kernel locks.

🌐

Multi-Library Support

Probes attach by inode, not PID. Once a libssl.so is probed, every process using that library is captured — including containers sharing the same base image layer.

🚨

Protocol Detection

Auto-detects HTTP/1.1, HTTP/2 (binary framing + HPACK), gRPC, and GraphQL. Non-HTTP protocols (database wire protocols, AMQP) are logged but not forwarded.

Anticipated Questions

Frequently Asked Questions

The most likely questions from engineering, security, and leadership — with concise, defensible answers.

Security & Risk
▶ Does this require privileged containers? Isn't that a security risk?

No, it does NOT require privileged mode. On kernel 5.8+ (which covers all modern distributions), the sniffer runs with privileged: false and uses fine-grained Linux capabilities instead:

  • CAP_BPF — load BPF programs
  • CAP_PERFMON — access perf events
  • CAP_SYS_PTRACE — read /proc/*/maps
  • CAP_DAC_READ_SEARCH — read across container boundaries

All other capabilities are explicitly dropped. The root filesystem is read-only. This follows the principle of least privilege — far less exposure than a privileged sidecar proxy.

▶ Can the sniffer read sensitive data like passwords and tokens?

It captures full plaintext HTTP including headers and body payloads, but sensitive data is automatically redacted before it leaves the pod. The built-in redactor masks:

  • Authorization, Cookie, Set-Cookie headers
  • API keys, Bearer tokens, Basic auth credentials
  • PCI-DSS patterns (credit card numbers) in request/response bodies

Additional headers can be added to the redaction list via configuration. The Reconstructor only needs URL patterns, methods, status codes, and header names — not the actual secret values.

▶ Where does captured data go? Does it leave our infrastructure?

The data flow is:

  1. Sniffer → Reconstructor — stays within your Kubernetes cluster (pod-to-pod HTTP)
  2. Reconstructor → Invicti Platform — only the reconstructed OpenAPI specification is uploaded, not raw traffic. The spec contains endpoint paths, methods, parameter names, and response schemas — no actual request/response payloads.

For fully air-gapped environments, the Reconstructor can run in offline mode with specs exported locally.

▶ Could an attacker exploit the eBPF sniffer to intercept traffic?

The sniffer is read-only — it observes traffic but cannot modify, inject, or redirect it. eBPF uprobes are a passive tracing mechanism. Additionally:

  • The sniffer pod uses readOnlyRootFilesystem: true
  • NetworkPolicy restricts egress to only DNS and the Reconstructor endpoint
  • RBAC grants only read access to node and pod metadata
  • Self-traffic is excluded (the sniffer's own PID is filtered at the BPF level)
Performance & Reliability
▶ What's the performance impact on production workloads?

Negligible. The eBPF program runs in kernel space with per-CPU lock-free buffers, so there's no contention or locking. Measured overhead:

  • CPU: ~100m (0.1 core) baseline, up to 500m under heavy load
  • Memory: 128Mi steady state, up to 512Mi during BCC compilation at startup
  • Latency added to requests: Zero. The uprobes execute asynchronously — the application doesn't wait for the sniffer.

Compare this to Istio sidecars which add 2-10ms latency per request and ~100Mi memory per pod.

▶ What happens if the sniffer crashes? Does it affect applications?

Applications are completely unaffected. When the sniffer pod stops, the kernel automatically detaches all uprobes. There is no sidecar, no iptables rule, no proxy in the data path. Applications continue to operate normally with zero impact. When the sniffer restarts, it reattaches probes and resumes capturing.

A PodDisruptionBudget ensures at least one sniffer remains running during rolling updates.

▶ Can it handle high-throughput environments (10K+ requests/sec)?

Yes. The architecture is designed for high throughput:

  • Per-CPU perf buffers — no cross-CPU locking, scales with core count
  • Configurable buffer size — tune perf_buffer_pages for your traffic volume
  • Port filtering — only capture traffic on ports you care about (e.g., 443, 8443)
  • Process filtering — filter by PID, UID, or process name to reduce noise

If the perf buffer overflows under extreme load, events are dropped (not queued) — the application is never backpressured. Lost events are tracked via the sniffer_events_lost_total Prometheus metric.

No Helm values to customize for basic deployment. No application restarts needed. Probes attach within seconds of pod startup.

▶ Does it work with EKS, GKE, AKS, and managed Kubernetes?

Yes. It works on any Kubernetes distribution that runs Linux nodes with kernel 4.18+:

  • AWS EKS — tested on Amazon Linux 2 and Bottlerocket nodes
  • GKE — works on Container-Optimized OS and Ubuntu nodes
  • AKS — works on Ubuntu-based node pools
  • K3s / RKE2 — tested extensively (this demo runs on K3s)
  • OpenShift — requires SCC (Security Context Constraints) allowing BPF capabilities

The only requirement is that nodes run Linux and allow BPF program loading (most managed K8s providers permit this by default).

▶ How do we monitor the sniffer itself?

Built-in observability on every pod:

  • Prometheus metrics (:9090/metrics) — events captured, events lost, latency histograms, probe count, correlator state, output circuit breaker status
  • Health endpoint (:8080/healthz) — JSON with event counters, uptime, probe status
  • Readiness probe (:8080/readyz) — returns 503 until BPF probes are attached
  • Structured JSON logs — every telemetry send is logged with method, path, status code, and running total

Key alert: sniffer_events_lost_total > 0 means the perf buffer overflowed — increase perf_buffer_pages or add port filters.

▶ What about new containers or services that start after the sniffer?

The sniffer discovers SSL libraries by inode, not by PID. If a new container starts using the same base image (and therefore the same libssl.so inode), the existing probes automatically capture its traffic — no rescan needed.

For containers with a completely new SSL library (different image), the sniffer supports periodic rescanning via SNIFFER_DISCOVERY_RESCAN_INTERVAL (in seconds). Alternatively, restarting the sniffer pod triggers a full rescan in seconds.

Technical Edge Cases
▶ Does it capture internal service-to-service traffic (east-west)?

Only if that traffic is TLS-encrypted. The sniffer hooks into SSL library functions. If service-to-service communication uses plain HTTP (common inside a cluster), the sniffer won't see it because SSL_read/SSL_write are never called.

For capturing plain HTTP east-west traffic, use the TAP plugin (pcap-based) or Istio integration instead. The eBPF sniffer is specifically designed for TLS-encrypted traffic that other approaches can't see.

▶ What about mutual TLS (mTLS) and service mesh traffic?

Yes, mTLS is fully supported. mTLS uses the same SSL_read/SSL_write functions as regular TLS — the only difference is that both sides present certificates. The eBPF probes capture the plaintext regardless of whether it's one-way or mutual TLS.

For Istio mTLS specifically, we also offer the Envoy WASM integration which captures traffic at the proxy level. The eBPF sniffer can complement this by catching traffic from workloads that bypass the Envoy sidecar.

▶ Can it capture traffic from statically-linked binaries?

Yes, with some nuance:

  • Go binaries — Go statically links crypto/tls. The sniffer discovers Go binaries by scanning /proc for known names (traefik, envoy, consul, etc.) and parsing their ELF symbols to find (*Conn).Read/(*Conn).Write offsets.
  • Rust binaries — Same approach for rustls-based binaries.
  • C/C++ static OpenSSL — If OpenSSL is statically compiled into a binary (rare but possible), the sniffer can attach via --extra-lib openssl:/path/to/binary.
▶ How accurate is the reconstructed OpenAPI spec?

The spec reflects what was actually observed in traffic — which is both its strength and limitation:

  • Strength: Every endpoint in the spec was genuinely called in production. No false positives from dead code or unused routes.
  • Limitation: Endpoints that are never called during the observation period won't appear. Rare error paths or admin endpoints may be missing.

For maximum coverage, run the sniffer for at least one full business cycle (typically 1-2 weeks) to capture seasonal and scheduled API patterns.

▶ Does it work with HTTP/2 and gRPC?

Yes. The protocol detector identifies HTTP/2 binary frames (magic prefix PRI * HTTP/2.0 and frame headers). HPACK-compressed headers are decoded. gRPC (which runs over HTTP/2) is detected and parsed including the :path, :method, and content-type: application/grpc headers.

The correlator handles HTTP/2's multiplexed nature by accumulating response HEADERS and DATA frames before emitting a complete telemetry event.

Business & Strategy
▶ Why eBPF instead of improving our existing gateway-based approach?

Gateway-based capture only sees traffic that flows through the gateway. It misses:

  • Service-to-service HTTPS calls that go directly between pods
  • Background jobs and cron tasks that make API calls without going through ingress
  • Third-party SDK calls to external APIs (Stripe, AWS, etc.)
  • Services that bypass the gateway (debug endpoints, admin APIs, health checks)

eBPF sees everything — it operates at the kernel level, below all application routing. The gateway approach and eBPF approach are complementary: gateways for traffic you control, eBPF for traffic you didn't know about.

▶ What's the competitive landscape? Who else does eBPF-based API discovery?

The eBPF API discovery space is emerging:

  • Cilium/Hubble — focuses on network observability (L3/L4), not HTTP-level API discovery from encrypted traffic
  • Pixie (New Relic) — eBPF-based observability but focused on APM, not API security or OpenAPI generation
  • Akita Software (acquired by Postman) — traffic-based API spec generation but uses pcap, not eBPF uprobe interception of TLS

Our differentiator: purpose-built for API security with native integration into the Invicti scanning platform. Discover an API → automatically scan it for OWASP API Top 10 vulnerabilities — closed loop.
